# Архитектура Spider

Обзор компонентов, потоков данных и крипто-протокола.

└──────────────┘                       └───────────────┬─────────────────┘
                                                       │
                          ┌────────────────────────────┴───────────────┐
                          │            Go-сервер (spider-server)        │
                          │  ┌─────────┐  ┌──────┐  ┌──────────┐        │
                          │  │  api    │─▶│ hub  │  │ commands │        │

                          │       │          │           │              │
                          │  ┌────▼──────────▼───────────▼─────┐        │

## Поток регистрации устройства (enrollment)

1. **Админ** создаёт в панели одноразовый enrollment-токен → `POST /admin/enrollments`.
   Сервер генерирует: случайный токен, симметричный ключ сессии (32 байта),
   эфемерную пару X25519 сервера.
2. **Админ** даёт команду установки агенту (через панель: токен виден один раз).
3. **Агент** при первом запуске показывает предупреждение, запрашивает
   подтверждение, затем `POST /agent/enroll` с токеном + сист.инфо.
4. **Сервер** атомарно гасит токен, генерирует `device_id`, сохраняет устройство
   с пришедшей сист.инфо, возвращает `device_id` + симметричный ключ.
5. **Агент** сохраняет state (`device_id` + ключ) в `spider-state.toml`.

> Регистрация идёт по **явному токену**, который админ заводит вручную. Это
> надёжнее «отпечатка системы» (железо апгрейдится — отпечаток плывёт) и
> точно отражает, какие машины в парке.

## Поток выполнения команды

1. Админ отправляет команду: `POST /admin/devices/:id/commands`.
2. `commands.Dispatcher` ставит её в очередь (SQLite, status=queued).
3. Если устройство онлайн (WS) — команда сразу шифруется и отправляется.
   Иначе — остаётся queued и забирается при следующем long-poll/подключении.
4. **Агент** расшифровывает, выполняет через системный shell (`sh -c` / `cmd /C`)
   с таймаутом, шифрует результат (stdout/stderr/exit) и отправляет обратно.
5. Сервер сохраняет результат, эмитит WS-событие → панель обновляется live.

## Крипто-протокол (поверх TLS)

Уровневая защита (defense-in-depth): TLS терминирует Caddy, а поверх — прикладной
слой, чтобы защитить коммерческую тайну даже при компрометации сертификата/прокси.

- **ECDH**: X25519 для вывода общего секрета.
- **KDF**: HKDF-SHA256 → 32-байтный ключ AES-256.
- **AEAD**: AES-256-GCM, nonce 12 байт, случайный для каждого сообщения.
- **Формат wire**: `Envelope { type, data: base64(Encrypt(json(payload))) }` —
  единый для WS и long-poll (DRY).

Подробности — в `server/internal/crypto/crypto.go` и `client/src/crypto.rs`.

## Транспорт

| Режим     | Когда               | Механика                                   |
|-----------|---------------------|--------------------------------------------|
| WebSocket | Основной            | Постоянное WSS-соединение, heartbeat 60с   |
| Long-poll | Fallback (WS упал)  | GET /agent/connect держит до N сек, batch  |

Оба режима используют общий формат Envelope и общий обработчик команд.
Автопереключение — в `client/src/transport.rs`.
